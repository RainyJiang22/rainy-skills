# 工程与规范、第三方库、可交付物

## 工程与规范

1. **目录结构**：与「合理分层」一致；lib/ 下 app / core / domain（可选）/ data / features / shared；core 放网络、存储、DI、错误；features 内按功能分子目录。
2. **命名**：遵循 Effective Dart；类/文件/变量一致；公共 API 清晰命名。
3. **分析选项**：analysis_options.yaml，推荐 lint；CI 跑 dart analyze 与测试。
4. **依赖管理**：pubspec 版本收敛；第三方库有选型理由与文档；大版本升级有兼容评估与回归。
5. **资源与国际化**：图片/字体按模块或类型归类；文案走 l10n/ARB，禁止硬编码用户可见文案。

## 第三方库使用原则

1. **选型**：pub.dev 评分高、维护活跃、与 Flutter/Dart 版本兼容；避免冷门或弃维护库。
2. **封装**：对外统一接口，内部可替换；业务不直接依赖第三方类型（如 Dio Response）。
3. **版本**：主库版本固定或范围可控；冲突时优先升级或替换，避免 override 滥用。
4. **文档与注释**：新库在 README/架构文档说明用途、封装位置、替换策略；复杂用法加注释或示例。

## 可交付物（架构落地建议产出）

1. **架构图**：分层图 + 模块/依赖关系图（Mermaid 或图示）。
2. **目录与模块说明**：各目录职责、核心类与接口列表。
3. **UI 与主题**：Material 3 主题配置；AppLoading/AppEmpty/AppError 等通用组件。
4. **网络/存储/路由示例**：各层 1～2 个完整调用链示例（UI → API/DB）。
5. **状态管理约定**：选型说明 + 创建/注入/释放约定。
6. **README 或 CONTRIBUTING**：新人接入步骤、规范链接、常用命令（分析、测试、运行）。

# Flutter 企业级 Code Review 技能

## 目标

1. **整体项目自动 Code Review 工作流**：从确定范围 → 制定计划 → 按模块执行 → 输出报告，形成可重复、可自动化的流程。
2. **保证代码高效、健壮、可扩展**：评审维度明确覆盖性能与资源（高效）、异常与边界（健壮）、架构与扩展点（可扩展）。
3. **合理的评审计划**：对指定项目区分模块，按模块制定并执行 Review，避免一次性全量、无重点的审查。

执行原则：**只读分析，只输出建议与最优解，不直接修改项目代码**。是否采纳由人工或后续任务决定。

---

## 一、整体自动 Code Review 工作流

按以下四步执行，形成闭环。

### 1. 确定评审范围与目标

- **范围**：全项目 / 指定目录（如 `lib/`）/ 指定模块列表 / PR 变更文件。
- **目标**：常规质量巡检 / 上线前评审 / 重构前评估 / 新人合入前规范校验 / 专项（性能、Crash、可扩展性）。
- **输出**：评审范围说明（路径、模块或文件列表）、本次侧重点（效率 / 健壮性 / 可扩展性 / 全面）。

### 2. 制定评审计划（按模块区分）

- 识别项目模块：以 `lib/` 下的一级子目录或业务域目录为候选模块（如 `common/`、`feature_a/`、`core/`、`ui/`）。
- 区分模块类型并排期：
  - **核心/基础模块**（如 `core/`、`common/`、网络/状态基础设施）：优先评审，影响面大。
  - **业务/功能模块**（如各 feature、业务 UI）：按依赖关系或业务重要性排序。
  - **支撑模块**（如 `utils/`、`widgets/`）：可与依赖它们的业务模块同批或稍后。
- 输出：**评审计划表**（模块名、路径、建议顺序、本批侧重点），便于分阶段执行。

详见 [references/module-review-plan.md](references/module-review-plan.md)。

### 3. 按模块执行评审

- 对计划表中的每个模块依次执行「检查清单」中的项，不修改代码，只记录通过/不通过及问题与建议。
- 单模块内按顺序检查：基础规范（命名、文件大小、魔法值）→ 架构与依赖 → 效率与资源 → 健壮性与异常 → 可扩展性与可维护性。
- 每完成一个模块即输出该模块的**建议报告**（建议位置 + 最优解方案 + 优先级 P0/P1/P2）。

### 4. 汇总与输出

- 全部模块完成后，输出**项目级汇总**：各模块问题数、P0/P1/P2 分布、修复优先级建议。
- 报告格式与示例见 [references/review-output.md](references/review-output.md)。

## 二、评审维度：高效、健壮、可扩展

### 高效（Efficiency）

- **资源**：无多余全局单例、大列表/大对象有懒加载或分页；Controller/Stream 在 dispose 中正确释放。
- **性能**：避免在 build 中做重计算或同步 I/O；列表使用 ListView.builder 等懒加载；图片/资源有适当缓存或压缩。
- **结构**：无重复请求、重复计算；可复用的逻辑提取到工具或 Service，避免重复代码导致冗余执行。

### 健壮（Robustness）

- **异常与边界**：所有异步入口（async、Future、Stream、Timer）有 try/catch 或 .catchError；不可信输入（网络、存储、用户输入）做校验与兜底。
- **空安全与类型**：合理使用 null safety，避免强制 `!` 滥用；外部数据解析有异常捕获与默认值。
- **生命周期与资源**：订阅、Controller、Stream 在 onClose/dispose 中取消或释放，避免泄漏与野回调。

### 可扩展（Extensibility）

- **模块边界**：按功能划分模块，模块间通过接口/事件/依赖注入通信，避免跨模块直接依赖实现类。
- **扩展点**：新功能优先通过扩展既有模块或新增子模块完成；公共能力沉淀到 common/ 或 core/，避免复制粘贴。
- **可测试与可替换**：业务逻辑可单测；网络/存储等通过接口注入便于 mock；避免在业务类中直接 static 或 new 具体实现。

完整检查项见 [references/checklist.md](references/checklist.md)。

## 三、分模块评审计划（制定与执行）

### 如何区分模块

- 以 `lib/` 下的一级子目录为候选（如 `lib/feature_login/`、`lib/core/`、`lib/common/`）。
- 若项目按业务域在二级目录划分（如 `lib/ui/feature_a/`），则以该业务域目录为「模块」。
- 每个模块应有相对清晰的职责边界；若某目录仅为散落文件，可合并到「其他」或按父目录归类。

### 建议评审顺序

1. **第一轮**：核心/基础模块（如 `core/`、`common/`、网络层、状态管理基础设施）。
2. **第二轮**：被多处依赖的业务模块或主流程模块。
3. **第三轮**：其余业务模块与支撑模块。

同一轮内可按目录名字母序或按依赖关系排序；有明确上线/重构目标的，可把相关模块提前。

### 单模块内执行顺序

对每个模块按 [references/checklist.md](references/checklist.md) 中的顺序执行：基础规范 → 架构与依赖 → 效率 → 健壮性 → 可扩展性。每项只记录结果与建议，不改代码。

### 输出要求（每模块 + 最终汇总）

- **每模块**：建议位置（文件:行或类/方法名）、问题简述、最优解优化方案、优先级（P0/P1/P2）。
- **最终**：汇总表（模块名、问题数、P0/P1/P2 分布）、整体修复优先级建议。

详见 [references/review-output.md](references/review-output.md) 与 [references/module-review-plan.md](references/module-review-plan.md)。

## 四、与 CI/脚本的衔接（自动化入口）

- **评审计划**：可用脚本扫描 `lib/` 得到候选模块列表，再按约定规则（目录名或配置文件）区分核心/业务/支撑，生成「评审计划表」。
- **单次评审执行**：由人工或 CI 指定「本次评审模块」或「本次评审文件列表」，按本技能的工作流与检查清单执行并输出报告。
- **脚本辅助**：如需要统计单文件行数、大文件列表，可结合项目内或技能内 scripts（若有）做辅助；核心评审逻辑仍以本技能描述的流程与清单为准。

# 评审检查清单（高效 / 健壮 / 可扩展）

按模块执行时，依下表逐项检查，只记录通过/不通过及建议，不修改代码。

---

## 一、基础规范

| 检查项 | 说明 | 不通过时建议 |
|--------|------|--------------|
| 命名与风格 | 遵循 Effective Dart 与项目 analysis_options；类/文件/变量命名一致 | 统一命名风格，补充/收紧 analysis_options |
| 文件大小 | 单文件建议 ≤350 行（可配置） | 拆分为多文件或抽取 mixin/extension |
| 魔法值 | 数字、字符串字面量有业务含义的提取为常量或枚举；≥2 次出现的必须提取 | 提取到常量/枚举，放 define/ 或文件顶部 |
| 目录与分层 | lib/ 下按职责分层，依赖方向清晰（如 View→Controller→Provider） | 调整目录或依赖，禁止反向依赖 |

---

## 二、架构与依赖

| 检查项 | 说明 | 不通过时建议 |
|--------|------|--------------|
| 分层职责 | View 不直接调网络/存储；Controller 不直接写 UI 逻辑；Provider 只做请求/序列化 | 把越界逻辑下沉或上移到正确层 |
| 依赖方向 | 无循环依赖；底层不依赖上层 | 抽接口、事件或依赖注入打破循环 |
| 模块边界 | 模块间通过接口/事件/公共 API 通信，不直接依赖实现类 | 引入接口或事件，隐藏实现 |

---

## 三、高效（Efficiency）

| 检查项 | 说明 | 不通过时建议 |
|--------|------|--------------|
| 资源释放 | Controller/Stream/Subscription 在 onClose/dispose 中取消或释放 | 在 onClose/dispose 中统一取消与关闭 |
| 列表与大数据 | 长列表用 ListView.builder 等懒加载；大对象/大集合有分页或懒加载 | 改为懒加载/分页，避免一次性加载 |
| build 内无重逻辑 | build 中不做同步 I/O、重计算或重复请求 | 将逻辑移到 init/onReady 或异步加载后 setState |
| 重复请求/计算 | 相同请求或计算不重复触发；可复用逻辑提取到 Service/Utils | 加缓存、防抖或抽取公共方法 |
| 图片与资源 | 图片有缓存或压缩策略；大资源不阻塞主流程 | 使用 cached_network_image 或等价方案，异步加载 |

---

## 四、健壮（Robustness）

| 检查项 | 说明 | 不通过时建议 |
|--------|------|--------------|
| 异步兜底 | 所有 async/Future/Stream/Timer 有 try/catch 或 .catchError | 在 async 内或链式调用加 catch，打日志并统一错误处理 |
| 不可信输入 | 网络、存储、用户输入做校验与兜底（空、类型、范围） | 校验 + 默认值/早期 return，避免 NPE 与异常 |
| 空安全 | 少用强制 `!`；可空类型有 null 处理 | 用 `?`、`??`、条件判断替代不必要的 `!` |
| 资源与生命周期 | 无订阅/Controller 泄漏；dispose 后不再回调 | 在 onClose/dispose 取消订阅、置空引用 |
| 错误与日志 | 捕获后打日志；不吞掉异常；release 不暴露敏感栈 | 统一错误处理与日志，区分 debug/release |

---

## 五、可扩展（Extensibility）

| 检查项 | 说明 | 不通过时建议 |
|--------|------|--------------|
| 扩展点 | 新功能能通过扩展既有模块或新增子模块完成，而不是大改核心代码 | 用策略/工厂/插件化或子模块封装新能力 |
| 公共能力沉淀 | 多处使用的逻辑/组件在 common/ 或 core/，无复制粘贴 | 抽取到公共模块，通过参数或配置复用 |
| 可测试 | 业务逻辑可单测；网络/存储等通过接口注入便于 mock | 依赖接口注入，避免 static/new 具体实现 |
| 配置与策略 | 超时、重试、开关等可配置，不写死 | 从配置或环境读取，提供默认值 |

---

## 六、网络与状态（若适用）

| 检查项 | 说明 | 不通过时建议 |
|--------|------|--------------|
| 网络统一入口 | 所有请求经统一封装（如 BaseHttp、Dio 封装），不在 UI/Controller 裸调 | 统一走一层 Provider/Repository，封装超时与重试 |
| 错误与重试 | 统一错误类型与重试策略，不阻塞主流程 | 定义 ApiException，重试可配置且带退避 |
| 状态管理一致 | 同一模块内状态管理方式一致（如 GetX/Provider/Riverpod 择一） | 统一方案，Controller 生命周期与页面/模块绑定 |
| 依赖注入 | 通过 Get.put/lazyPut 或 DI 显式注入，避免深层 Get.find() 未注册 | 在合适层级注册，通过构造函数或 find 注入 |

---

## 优先级定义

- **P0**：必改（影响正确性、稳定性、安全或严重违反架构）。
- **P1**：应改（影响健壮性、可维护性或性能，建议本迭代处理）。
- **P2**：建议改（规范、可读性、可扩展性优化，可排期）。

每个不通过项在「建议报告」中标注 P0/P1/P2，便于排期修复。

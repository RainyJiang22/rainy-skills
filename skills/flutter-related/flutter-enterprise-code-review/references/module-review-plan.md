# 分模块评审计划

## 1. 模块识别

### 1.1 候选模块来源

- **一级目录**：`lib/` 下每个子目录视为一个候选模块（如 `lib/core/`、`lib/common/`、`lib/features/`）。
- **业务域目录**：若结构为 `lib/ui/feature_xxx/` 或 `lib/domain/yyy/`，则以 `feature_xxx`、`yyy` 为模块名，路径为 `lib/ui/feature_xxx`、`lib/domain/yyy`。
- **根文件**：`lib/` 根目录下的单文件（如 `main.dart`、`app.dart`）可合并为「入口/应用」模块或单独列为「根级」。

### 1.2 模块类型与优先级

| 类型 | 说明 | 建议评审顺序 |
|------|------|--------------|
| 核心/基础 | 网络、状态管理基类、路由、通用错误处理、公共 model | 第一轮 |
| 公共层 | common/、utils/、widgets/、define/ | 第一轮或第二轮前 |
| 业务模块 | 各 feature、业务 UI、领域服务 | 第二轮起，按依赖或业务重要性 |
| 支撑/其他 | 单点工具、脚本、临时目录 | 最后或与依赖方同批 |

### 1.3 依赖关系与顺序

- 被多处依赖的模块（如 `core/`、`common/`）优先评审。
- 若模块 A 依赖模块 B，则 B 建议先于 A 评审。
- 无明确依赖时，按「核心 → 公共 → 业务 → 支撑」排序。

---

## 2. 评审计划表格式

制定计划时输出如下结构的表格，便于按批执行：

| 顺序 | 模块名 | 路径 | 类型 | 本批侧重点 |
|------|--------|------|------|------------|
| 1 | core | lib/core/ | 核心 | 架构、健壮性、可扩展 |
| 2 | common | lib/common/ | 公共 | 规范、效率、可复用 |
| 3 | feature_a | lib/features/feature_a/ | 业务 | 健壮性、依赖方向 |
| … | … | … | … | … |

**本批侧重点**可填：全面 / 效率 / 健壮性 / 可扩展性 / 规范与命名。

---

## 3. 分阶段执行建议

- **阶段一**：只做核心 + 公共模块，输出其建议报告与 P0 问题列表。
- **阶段二**：按计划表做业务模块，每模块输出建议报告。
- **阶段三**：汇总全项目、排修复优先级，输出汇总表与后续改进建议。

若时间有限，可仅做「阶段一 + 高风险业务模块」，在计划表中标注「本轮必审」与「可延后」。

---

## 4. 使用脚本辅助（可选）

在项目根目录列出 `lib/` 下子目录可作为候选模块：

```bash
# 列出 lib 下一级目录（候选模块）
find lib -maxdepth 1 -type d ! -path lib | sed 's|lib/||' | sort
```

结合项目实际结构调整「模块类型」与「建议评审顺序」后，填入评审计划表即可。

# 评审输出格式

## 1. 单模块建议报告

每完成一个模块的评审，输出如下格式（只输出建议，不修改代码）。

### 1.1 模块标题

```text
[模块] <模块名>（<路径>）
```

### 1.2 问题表格

| 序号 | 建议位置 | 问题简述 | 最优解优化方案 | 优先级 |
|------|----------|----------|----------------|--------|
| 1 | 文件路径:行号 或 类名.方法名 | 一句话描述 | 具体改法、示例代码或步骤 | P0/P1/P2 |
| 2 | … | … | … | … |

- **建议位置**：精确到文件路径，可选行号或类名/方法名。
- **最优解优化方案**：推荐的具体改法，便于人工实施。
- **优先级**：P0 必改 / P1 应改 / P2 建议。

### 1.3 示例

```text
[模块] core（lib/core/）

| 序号 | 建议位置 | 问题简述 | 最优解优化方案 | 优先级 |
|------|----------|----------|----------------|--------|
| 1 | http_client.dart:45 | 请求超时硬编码 5000 | 提取为 const kTimeoutMs = 5000 或从配置读取 | P2 |
| 2 | auth_controller.dart login() | 无 try/catch | 在 async 方法内包 try/catch，catch 中打日志并调用统一错误处理 | P0 |
```

## 2. 项目级汇总

全部模块评审完成后，输出汇总表与修复优先级建议。

### 2.1 汇总表

| 模块名 | 路径 | 问题总数 | P0 | P1 | P2 |
|--------|------|----------|-----|-----|-----|
| core | lib/core/ | 3 | 1 | 1 | 1 |
| common | lib/common/ | 2 | 0 | 1 | 1 |
| … | … | … | … | … | … |

### 2.2 修复优先级建议

按以下顺序建议修复顺序（示例）：

1. **P0 全量**：先修复所有 P0，保证稳定性与正确性。
2. **核心/公共模块 P1**：再修复 core、common 的 P1，降低后续改动成本。
3. **业务模块 P1**：按依赖或业务重要性排序。
4. **P2**：按模块或迭代排期，可与新需求一起做。

---

## 3. 评审计划表（制定阶段输出）

在「制定评审计划」步骤输出，格式见 [module-review-plan.md](module-review-plan.md) 第 2 节，便于分阶段执行与跟踪。

---

## 4. 原则重申

- 所有输出为**建议**，不直接修改项目代码。
- 建议位置与最优解尽量具体，便于开发直接实施或排期。
- 优先级用于排期，P0 必须在本迭代或上线前处理。
